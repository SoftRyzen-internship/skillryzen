- Додаємо логотип компанії (можливо десь біля аватару). Оскільки менеджерів в кабінеті компанії може бути декілька, то потрібно показувати як аватар менеджера і його налаштування, так і логотип компанії, до якої належить кабінет.

**VIII .Сайтбар**

- **Включає слідуючі розділи:**
  - Дашборд. Показує в основній частині аналітику та статистику по кандидатам, які належать до компанії, їх активність і т.д.
  - Сертифікація. Даж можливість налаштувати різні види сертифікації (підсумкові/залікові тести). В МВП-1 буде лише фінальний тест. ТОму тут має бути можливість налаштувати цей тест - його тривалість (наприклад 2 год), вказати модулі та кількість питань у них, а також обрати відсоткове співвідношення важкості питань по кожній категорії (легких, середніх, важких).
  - Студенти/кандидати. Розділ веде на сторінку для роботи з учасниками. Має бути фільтрація учасників в списку, додавання нових учасників, можливість редагування контактних даних учасника та групи його приналежності (DS-21, FE-42) та вигрузка даних по учасникам в певних форматах - pdf, excel.
  - Invite module. Тут будуть усі налаштування для можливості створення урлів та qr-кодів для запрошення студентів/кандидатів до компанії.

**IX. Фінальний тест**

1. В МВП-1 ми концентруємся над функціоналом фінального тесту. Це буде тест, який можна пройти лише 1 раз з можливістю перездачі (1 або 2 рази) через певний проміжок часу.

2. Тест буде для розробників. Він створюється менеджером в кабінеті компанії, де вказуються модулі, які будуть включені в тест (HTML, CSS, JS і т.д.), кількість та важкість питань по кожному модулю, тривалість тесту (1 год, 2 год, 60хв), прохідний бал
   Перед стартом теста - кандидату буде показано опис тесту та умови (час, кількість питань, прохідний бал і т.д.) і, можливо, статистика (який середній бал вірних відповідей, кількість учасників, які його пройшли).

3. Сам тест в МВП-1 буде складатись зі слідуючого типу питань - 1 з 4 (одна вірна відповідь з 4 варіантів). Потім будемо розширювати - декілька вірних зі списку, дописати слово або частину коду в питанні, заповнити пропуск, обрати true | false.

4. Коли фінальний тест стартував - вийти з нього не буде можливості. Тест обмежений часом, тому потрібен таймер (наприклад, 1,5 години). Також, бажано вказувати помітки по кількості питань. Наприклад, якщо питань 50 - то внизу під текстом робити 50 колб (штрихів). І по ходу тесту забарвлювати їх в залежності від пройдених питань та вірних відповідей. Наприклад, кандидат пройшов 10 питань з 50 - то ми заливаємо цих 10 питань зеленим кольором при умові, що вони були вірні, або червоним - там де була помилка. Решта 40 штрихів - тусклі (ще не пройдені)
5. Якщо відповідь вірна або не вірна - зразу вказуємо про це кандидату і йдемо далі, не пояснюючи чому і що.
6. **При завершенні фінального тесту показуємо кандидату його результат:**
   - Загальна кількість і % вірних відповідей.
   - Кількість і % вірних відповідей по модулям.
   - Рекомендації по тесту: наприклад, кандидат найгірше відповів по JS - система сповіщає його, що потрібно більше уваги звернути на JS, а саме на розділ “Асинхронність”.
   - Кнопка провести ревю результатів. Але тут питання - можливо в фінальному тесті це і не потрібно, щоб вони не скрінили вірні відповіді.
   - Кнопка “Закінчити тест”

## Детальніше про структуру проєкту для розробників:

- **ui-kit/:**
  папка, в якій розміщені загальні компоненти для роботи з певною частиною додатку.
- **modules/:**
  папка, в якій розміщені модулі додатку. Модуль - це логічно зв'язаний набір компонентів та функцій для роботи з певною частиною додатку.

  - **common/:**

    - ця папка містить компоненти та інші ресурси, які використовуються в багатьох частинах проекту. Наприклад, тут можна зберігати загальні компоненти, які використовуються в декількох місцях.

  - **pages/:**

    - ця папка містить компоненти, які відповідають за відображення головної сторінки, дашборду, інших сторінок.

- **theme/:**

  ця папка містить константи глобальні, кольори, іконки, глобальні стилі

#### Prettier настройки для проєкту

**Опис кожної опції:**

- semi: true - вмикає автоматичне додавання крапки з комою в кінці рядка.

- trailingComma: 'es5' - встановлює, як Prettier повинен додавати кому після останнього елемента в об'єкті або масиві. В даному випадку, він додає кому після останнього елемента згідно зі стандартом ECMAScript 5.

- singleQuote: true - встановлює використання одинарних лапок для рядків.

- jsxSingleQuote: true - опція використовується для того, щоб змінити стиль форматування рядків у JSX-виразах. Використання jsxSingleQuote: true дозволяє використовувати одинарні лапки в рядках JSX замість подвійних лапок, що допомагає стандартизувати цей підхід до форматування.

- printWidth: 80 - встановлює максимальну ширину рядка. Якщо рядок перевищує це значення, Prettier спробує перенести його на новий рядок.

- tabWidth: 2 - встановлює кількість пробілів, які використовуються для відступів замість табуляції.

- endOfLine: 'auto' - встановлює символ нового рядка. Значення "auto" означає, що Prettier використовуватиме символ нового рядка, який знаходиться в кінці першого рядка в файлі.

#### Eslint Rules

**Опис кожної опції:**

- react/jsx-uses-react: 'off'

  - Це правило вимикає перевірку використання бібліотеки React у компонентах, де він не імпортується явно.

  - Це правило вимикає перевірку використання бібліотеки React в компонентах, де він не імпортується явно.

- @typescript-eslint/explicit-module-boundary-types: 'off'
  - Це правило вимикає перевірку на те, що всі функції та методи повинні мати явно визначений тип повернення в TypeScript коді. Оскільки в даному проекті він вимкнений, це означає, що не потрібно явно вказувати тип повернення функцій та методів.
